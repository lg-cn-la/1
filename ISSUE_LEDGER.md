# Sampora Issue Ledger

This ledger prevents historical issues from being rediscovered and repaired again without current evidence.

Use it before implementing any GPT, human, Codex, downloaded-report, or agent-generated issue list. The ledger is not proof by itself; it is a routing layer for status, owner, and required evidence.

## How To Use

1. If a new issue resembles an entry below, start from the ledger status instead of treating the issue as new.
2. Verify against current files, scripts, browser state, or zip entries.
3. Reclassify only with fresh evidence:
   - `[REGRESSION]` becomes `[CONFIRMED]` only when its current guard fails.
   - `[REMOVED]` stays no-repair/no-restore unless current evidence proves the issue has returned.
   - `[STRATEGY]` stays limited to docs, QA, packaging, or workflow policy.
4. If a repair changes source, reports, QA scripts, or zips, update the row's current state and last evidence.
5. Add new recurring issues here with a stable ID before they appear in multiple handoff entries.
